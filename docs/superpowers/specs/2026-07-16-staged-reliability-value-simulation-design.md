# 博弈预测看板：分阶段自动化、价值投注与模拟盈亏设计

## 1. 目标

在现有竞彩足球预测、平局预警、模拟方案、赛果结算、网站和邮件日报基础上，分阶段完成两类改进：

1. 让北京时间 12:15 至 18:00 的数据导入、预测、赔率刷新、结算和邮件发送形成可验证的自动闭环，不再发送日期错误或内容过期的日报。
2. 让系统每天从允许的竞彩玩法中自主选择具备正投注价值的模拟方案，使用保守的四分之一凯利公式分配金额，并持续展示每日与累计模拟盈亏。

系统仍是概率研究和模拟记账工具，不连接真实投注账户，不承诺盈利，也不会因为短期盈利自动提高预算。

## 2. 已确认约束

- 总体架构采用“Apps Script 调度、GitHub Actions 执行”。
- GitHub 的定时任务保留为后备路径，Apps Script 主动触发作为主要路径。
- Apps Script 是唯一邮件发送者；GitHub 邮件工作流保持关闭，防止重复发信。
- 数据未就绪时从北京时间 14:00 起每 10 分钟重试，最晚等待到 18:00。
- 18:00 仍未就绪时只发送一次异常报告，不附带旧方案。
- 每日模拟投入不超过 500 元。
- 每月模拟投入不超过 5000 元。
- 当月累计已实现亏损达到 5000 元后，停止当月新增模拟投注。
- 允许零投注日，未使用预算不滚存、不补投。
- 允许玩法仅包括胜平负、让球胜平负、进球数单关和最多 2 串 1。
- 不使用比分、半全场或 3 场以上串关。
- 单场全部相关模拟投入合计不超过 200 元。
- 串关每日模拟投入合计不超过 30 元。
- 模拟金额按 2 元向下取整。
- 网站标题从“世界杯每日预测看板”改为“博弈预测看板”。
- 邮件图片展示当天方案、前一天结算、最近 30 天每日盈亏和历史累计摘要；网站保留完整历史。

月度止损与月度投入上限同为 5000 元，因此止损不会比月度投入上限更早触发。两项规则仍分别记录和检查，方便以后独立调整。

## 3. 不在本次范围内

- 自动购买、代购、支付或连接任何真实投注账户。
- 承诺收益率、命中率或固定盈利。
- 为了每日都有方案而降低价值、数据质量或风险门槛。
- 使用赛后信息修改已锁定的赛前概率、赔率或金额。
- 将外部博彩市场赔率替换为国内竞彩方案赔率。

## 4. 总体架构

系统分为五个边界清晰的组件：

1. **Apps Script 调度器**：按北京时间判断当前阶段，触发 GitHub 工作流、轮询报告状态、去重并发送邮件。
2. **GitHub 自动化工作流**：运行数据导入、预测、市场刷新、赔率快照、模拟方案、赛果结算、模型评估、网站和日报图片构建。
3. **报告状态发布器**：生成机器可读的 `web/report-status.json`，作为 Apps Script 判断是否可发送的唯一依据。
4. **价值方案与模拟账本**：筛选正价值选项、分配模拟金额、锁定方案并按官方赛果结算。
5. **网站与邮件报告**：从同一份结构化输出构建完整网站和精简邮件图片，避免两个展示渠道口径不同。

组件只通过版本化 JSON/CSV 文件和 GitHub workflow dispatch 接口通信。Apps Script 不重新计算预测和盈亏，只负责调度、检查和发信。

## 5. 分阶段实施

### 5.1 第一阶段：自动运行与邮件可靠性

第一阶段不改变现有模型和方案筛选逻辑，只增加调度、状态、去重和失败处理。

- 新增报告状态文件及生成器。
- 让关键 GitHub 工作流支持幂等的 `workflow_dispatch`。
- 将 Apps Script 改为每 10 分钟运行一次，并按北京时间窗口执行动作。
- 增加同一天工作流触发去重、邮件去重和并发锁。
- 邮件只在状态文件满足完整性条件时发送。
- 增加 18:00 异常邮件和可追踪的失败原因。

### 5.2 第二阶段：价值投注与资金分配

第一阶段稳定后启用统一候选池、四分之一凯利金额、相关性限制、日/月预算和方案锁定。

- 先以影子模式生成新旧金额对照，不写入正式模拟盈亏。
- 对照通过自动测试和人工检查后，新方案才成为正式模拟账本输入。
- 策略切换只影响切换后的新方案，不重算历史记录。

### 5.3 第三阶段：报告、评估与守护式学习

- 统一每日与累计盈亏口径。
- 更新网站标题、完整历史表和邮件图片。
- 完善按玩法、联赛和模型版本的概率与经济指标。
- 候选模型继续采用影子评估、受控升级和可逆回滚。

## 6. Apps Script 调度设计

### 6.1 单一触发器

Apps Script 使用一个每 10 分钟触发器。代码内部使用 `Asia/Shanghai` 时区计算业务日期和阶段，不依赖浏览器或账号显示时区。

每次运行先取得 `LockService` 脚本锁；无法取得锁时直接退出，避免并发触发两封邮件或两个相同工作流。

### 6.2 时间窗口

- **12:15 以后**：若当天基础预测尚未就绪，触发 `daily-forecast.yml`。
- **13:30 以后**：若当天决策赔率快照或市场刷新尚未就绪，触发市场刷新工作流。
- **13:45 以后**：若尚未结算到前一天，触发 `noon-settlement.yml`。
- **14:00 至 18:00**：每 10 分钟读取状态；满足发送条件后立即发送一次日报。
- **18:00 以后**：当天仍未发送成功且状态不完整时，只发送一次异常报告。

每个动作使用“业务日期 + 动作名称”作为幂等键。Script Properties 保存最后触发时间，短时间内不会重复启动相同工作流。GitHub 工作流本身仍必须幂等，因为网络超时可能让 Apps Script 无法确认第一次请求是否成功。

### 6.3 GitHub 凭据

GitHub fine-grained token 只保存在 Apps Script Script Properties 中，不写入代码、日志、网页或邮件。

令牌只授予目标仓库的 Metadata 只读和 Actions 读写权限。状态文件从公开 GitHub Pages 读取，不扩大令牌权限。

### 6.4 Apps Script 状态

Script Properties 至少保存：

- `GITHUB_TOKEN`
- `LAST_FORECAST_DISPATCH_DATE`
- `LAST_REFRESH_DISPATCH_DATE`
- `LAST_SETTLEMENT_DISPATCH_DATE`
- `LAST_SENT_DATE`
- `LAST_SENT_IMAGE_SHA256`
- `LAST_FAILURE_NOTICE_DATE`

发送成功后才写入 `LAST_SENT_DATE` 和图片校验值。发送失败不能提前标记成功。

Apps Script 源码在仓库中保存版本化副本，线上项目与仓库版本必须标注相同版本号。

## 7. 报告状态契约

GitHub 每次成功重建报告后原子写入 `web/report-status.json`。至少包含：

```json
{
  "schema_version": 1,
  "report_date": "2026-07-16",
  "build_id": "20260716T142200Z-workflow-run-id",
  "generated_at_bjt": "2026-07-16T14:22:00+08:00",
  "decision_odds_at_bjt": "2026-07-16T13:30:00+08:00",
  "plan_locked_at_bjt": "2026-07-16T14:20:00+08:00",
  "settled_through": "2026-07-15",
  "forecast_ready": true,
  "decision_snapshot_ready": true,
  "settlement_ready": true,
  "plan_ready": true,
  "official_fixture_count": 12,
  "official_odds_count": 12,
  "official_odds_coverage_ratio": 1.0,
  "data_quality": "complete",
  "image_sha256": "hex-digest",
  "source_commit_sha": "git-sha-used-as-build-input",
  "source_status": {}
}
```

零场比赛或零投注方案是合法状态，必须明确写入零数量并将相应流程标记为成功，不能与抓取失败混淆。

### 7.1 邮件发送条件

日报只在以下条件全部满足时发送：

- `schema_version` 是受支持版本。
- `report_date` 等于北京时间当天。
- `forecast_ready`、`decision_snapshot_ready`、`settlement_ready` 和 `plan_ready` 全为 `true`。
- `settled_through` 不早于北京时间前一天。
- `generated_at_bjt` 晚于当天决策赔率快照。
- `plan_locked_at_bjt` 非空且不晚于 `generated_at_bjt`。
- `image_sha256` 非空，且与当天已发送校验值不同。
- 日报图片可下载、非空，Apps Script 对下载内容计算的 SHA-256 与状态文件一致。

任何条件不满足都视为“尚未就绪”，不能回退发送昨天的图片。

`build_id` 在单次构建开始时生成，并同时写入状态、网站和图片元数据。Apps Script 使用带 `build_id` 的防缓存请求读取状态与图片，避免 GitHub Pages 缓存造成新状态配旧图片。`source_commit_sha` 只记录本次构建使用的输入版本，不要求预知构建完成后自动生成的提交哈希。

## 8. GitHub 工作流设计

- 所有写入仓库的工作流共用 `sporttery-repository` 并发组，避免相互覆盖。
- 工作流始终从 `main` 最新状态开始，并在提交前重新确认目标业务日期。
- GitHub cron 保留为后备触发；workflow dispatch 是正常路径。
- 同一业务日期重复运行时，已完成步骤可以重算，但不能重复追加同一笔模拟方案或结算。
- 状态文件只在网站、图片和账本均成功生成后更新为就绪。
- 某个可选外部市场失败可以降级继续；国内竞彩赔率、方案锁定、账本或报告图片失败必须保持未就绪。
- `email-report.yml` 保持 `disabled_manually`，避免 SMTP 与 Apps Script 双重发送。

## 9. 数据来源与时点

### 9.1 国内竞彩数据

- 比赛列表、允许玩法、方案赔率和结算赔率优先使用竞彩网。
- 竞彩网不可用时，可使用中国足彩网或已验证的国内竞彩镜像作为备用。
- 每个方案必须保存具体来源、抓取时间和原始标识。
- 没有可验证国内竞彩赔率的选项只能研究展示，不能进入模拟方案。

### 9.2 外部分析数据

专业博彩网站、交易市场、球队官方信息和赛事组织方数据只用于概率分析、伤停、阵容、热度和赔率变化证据，不能替代国内竞彩方案赔率。

缺失信息不进行猜测填充。来源冲突时保留各来源和时间戳，降低数据质量；无法判定时观望。

### 9.3 赔率锁定

方案使用决策时点保存的国内竞彩赔率。邮件发送后，比赛、玩法、选择、金额和赔率不可修改。

后续赔率快照只用于 CLV、漂移和复盘，不覆盖历史方案。已开赛比赛不再写入赛前快照。

## 10. 价值候选池

### 10.1 允许玩法

- 胜平负单关
- 让球胜平负单关
- 进球数单关
- 2 串 1

比分、半全场和 3 场以上串关不进入候选池。

### 10.2 概率与价值

每个候选项分别保存：

- 原始模型概率
- 联赛校准概率
- 保守决策概率
- 国内竞彩去水市场概率
- 概率优势
- 十进制赔率
- 预期收益 `EV = conservative_probability * odds - 1`
- 数据质量等级
- 赔率波动等级

候选项必须同时满足：

- 国内竞彩赔率和比赛身份验证通过。
- 保守决策概率高于去水市场概率。
- `EV > 0`，并达到当前玩法配置的最低优势和最低 EV 门槛。
- 数据质量不低于中等。
- 比赛尚未开赛，且赔率属于允许的赛前时点。

现有更严格的玩法门槛继续生效，不因本设计降低。候选为空时生成正式的零投注方案和观望原因。

### 10.3 串关规则

- 只允许 2 串 1。
- 每一腿单独通过正价值和数据质量门槛。
- 两腿不能来自同一场比赛。
- 明显共享同一结果驱动因素或高度相关的选项不组合。
- 组合后的保守预期收益仍必须为正。
- 串关每日模拟投入合计不超过 30 元。

## 11. 四分之一凯利资金分配

对十进制赔率 `o`、净赔率 `b = o - 1`、保守概率 `p` 和 `q = 1 - p`：

```text
full_kelly = max(0, (b * p - q) / b)
raw_stake = reference_bankroll * full_kelly * 0.25
```

默认月度参考模拟资金为 5000 元。原始金额再应用以下限制：

- 高数据质量乘数为 `1.0`，中等数据质量乘数为 `0.6`，低质量直接排除。
- 决策赔率相对开盘赔率异常波动且原因未确认时乘数最多为 `0.75`。
- 已存在同场相关敞口时先合并计算，再执行单场 200 元上限。
- 单笔最终金额不得超过剩余日额度、剩余月额度、单场额度和玩法额度。
- 最终金额按 2 元向下取整；小于 2 元则不生成该笔方案。

组合排序优先考虑保守 EV、概率校准可靠性、数据质量和风险分散，不按名气、热门程度或单纯命中率排序。

## 12. 风险控制

- 每日新模拟投入总额不超过 500 元。
- 每月新模拟投入总额不超过 5000 元。
- 当月累计已实现净盈亏小于或等于 `-5000` 元时停止新增模拟投入。
- 单场全部玩法合计不超过 200 元。
- 串关每日合计不超过 30 元。
- 同一选择不会因同时属于主方案和平局预警而重复投入或重复计算利润。
- 未结算方案占用已投入额度，但不提前计入已实现盈利或亏损。
- 未使用预算不会转移到下一天或强制补投。
- 风险规则在候选排序之后再次统一检查，任何模块都不能绕过总预算。

## 13. 不可变模拟账本

每笔正式模拟方案使用稳定 `bet_id`，至少保存：

- 方案日期和锁定时间
- 国内竞彩比赛 ID、联赛、主客队和开赛时间
- 玩法、选择、让球或进球数参数
- 国内竞彩来源和锁定赔率
- 原始、校准、保守和市场概率
- 概率优势、EV、凯利比例和各风险乘数
- 模拟金额
- 方案版本和模型版本
- 状态：待结算、命中、未中、退款或异常
- 90 分钟赛果、结算来源和结算时间
- 返还、已实现盈亏和 CLV

方案采用追加和状态迁移，不删除旧记录，不用赛后赔率修改锁定赔率。重跑同一业务日期时通过稳定 `bet_id` 幂等更新，不能生成重复记录。

## 14. 结算口径

- 胜平负和让球胜平负按国内竞彩公布的 90 分钟常规时间口径结算。
- 加时赛和点球不改变 90 分钟玩法结果。
- 单关命中返还为 `stake * locked_odds`，未中返还为 0，盈亏为返还减投入。
- 2 串 1 全部有效腿命中时，返还为投入乘以各腿锁定赔率之积。
- 取消或无效腿按国内竞彩正式结算规则处理；确认退款的腿按赔率 `1.0` 处理。
- 赛果缺失或来源冲突时保持待结算，不猜测、不提前记账。
- 金额和累计指标保留两位小数。

## 15. 每日与累计盈亏

### 15.1 每日表

每个方案日期一行，展示：

- 投注笔数和已结算笔数
- 当日模拟投入
- 当日返还
- 当日已实现盈亏
- 当日 ROI
- 累计投入、累计返还和累计盈亏
- 累计最大回撤
- 待结算敞口

当日仍有待结算方案时明确标记，不把部分结算误写成最终日收益。

### 15.2 综合表

展示：

- 总投入、总返还、累计盈亏和 ROI
- 命中率、CLV、Brier、Log Loss 和校准误差
- 最大回撤、最长连续未中和当前连续未中
- 按玩法、联赛、模型版本和数据质量分组的独立表现
- 本月投入、本月已实现盈亏、剩余额度和止损状态

概率指标和经济指标分开展示，不能用短期盈利掩盖概率恶化。

## 16. 网站与邮件

### 16.1 网站

- 页面主标题改为“博弈预测看板”。
- 展示当天方案、观望原因、赔率来源与时点、前一天结算、完整每日盈亏和累计指标。
- 完整历史保留在网站，支持按玩法、联赛和状态查看。
- 页面顶部明确显示报告日期、北京时间生成时间、数据质量和结算截止日期。

### 16.2 邮件图片

一张日报图片依次展示：

1. 报告状态和数据时间。
2. 当天正式模拟方案或零投注观望结论。
3. 前一天逐笔结算摘要。
4. 最近 30 天每日盈亏表。
5. 历史累计摘要和各玩法表现。
6. 本月投入、剩余额度和止损状态。

完整历史通过网站链接查看，避免邮件图片无限增长。

### 16.3 邮件去重与异常邮件

- 正常日报使用日期和图片 SHA-256 双重去重。
- 同一天内容发生变化但正常日报已发送，不再自动补发；网站继续更新。
- 18:00 异常邮件每天最多一封，列出缺失步骤、最后成功时间和网站链接。
- 异常邮件不附带过期投注方案或旧日报图片。

## 17. 模型评估与守护式学习

- 保留现有原始概率、联赛校准概率、保守决策概率和市场公平概率四层记录。
- 单个联赛至少 30 场有效结算样本且时间外 Brier 改善后，才允许启用受限校准。
- 每日结算后更新指标；候选模型按固定周期训练，不因单场结果即时改权重。
- 候选模型先影子运行，与冠军模型使用相同后续样本比较。
- 晋级必须同时改善时间外概率指标，并保持正 CLV；ROI 只作辅助条件，不能成为唯一目标。
- 新规则至少需要 30 场完整样本，已知经典比赛只用于回归测试和解释。
- 某玩法或联赛出现持续概率恶化、负 CLV 或异常回撤时自动降额或暂停。
- 训练失败、模型文件损坏或指标不足时继续使用有效冠军模型；无有效冠军时回退基础模型。

## 18. 错误处理

- 国内竞彩比赛或赔率缺失：该候选不进入方案，报告记录缺失原因。
- 外部分析源失败：记录来源错误并降低数据质量；不影响可独立完成的基础预测。
- 队名无法可靠映射：隔离该比赛并记录人工映射需求，不猜测匹配。
- 赛果缺失：保持待结算并在下一次任务重试。
- GitHub dispatch 请求超时：Apps Script 保留幂等键，下一轮先读状态再决定是否重发。
- GitHub Actions 延迟：Apps Script 持续轮询至 18:00，不发送旧报告。
- 日报图片或状态文件不一致：视为未就绪。
- Gmail 发送失败：不写成功标记，下一轮重试。
- 18:00 仍失败：发送单一异常邮件并停止当天正常日报重试。

## 19. 测试

### 19.1 单元测试

- 北京时间窗口和跨日边界。
- 报告状态完整、缺失、过期和版本不支持。
- Apps Script workflow dispatch、邮件和异常通知幂等键。
- 去水概率、EV 和四分之一凯利计算。
- 2 元取整、单场 200、串关 30、每日 500、每月 5000 和月亏损 5000 限制。
- 中等/低数据质量和异常赔率波动处理。
- 同场相关敞口、主方案与预警去重。
- 单关、2 串 1、退款、待结算和 90 分钟口径。
- 每日与累计盈亏、ROI、回撤和连续未中。

### 19.2 集成测试

- Apps Script 测试模式触发 GitHub 测试工作流，但不发送真实邮件。
- 当天报告未就绪时不会发送昨天图片。
- 状态在 14:00 至 18:00 之间转为就绪后只发送一次。
- 18:00 仍未就绪时只发送一次异常邮件。
- 工作流重复运行不会重复方案、重复结算或重复累计盈亏。
- 零比赛和零投注日仍能生成合法状态、网站和邮件。
- 网站与邮件使用同一方案、结算和指标来源。

### 19.3 模型验证

- walk-forward 训练时间始终早于验证比赛。
- 候选模型没有足够影子样本时不能晋级。
- 概率指标恶化但短期 ROI 为正时不能晋级。
- 模型回滚不修改历史方案和历史概率。

## 20. 上线与回滚

1. 先发布状态文件和工作流幂等逻辑，不改变现有方案。
2. 部署 Apps Script 调度器测试模式，确认 dispatch、轮询和去重。
3. 启用每 10 分钟触发器，保留 GitHub cron 后备，继续关闭 GitHub 邮件工作流。
4. 观察至少一个完整业务日，确认正常日报和异常路径。
5. 影子运行新资金分配，核对金额、预算和账本。
6. 启用新模拟方案并更新网站与邮件。
7. 最后启用扩展评估和自动降额/暂停。

每一阶段均可独立回滚。第一阶段失败时可恢复当前北京时间 14:00 至 15:00 的单一 Apps Script 日触发器；第二阶段失败时继续使用旧方案生成逻辑；第三阶段失败时继续保留账本和基础指标。

## 21. 验收标准

- 连续业务日不再出现北京时间凌晨 5 点误发。
- 正常日报只在状态文件证明为当天完整报告时发送。
- 同一天不会收到重复正常日报或重复异常邮件。
- 14:00 未就绪时自动重试，最晚到 18:00。
- 18:00 未就绪时不发送旧方案。
- 所有正式模拟方案均有可验证国内竞彩赔率和赛前锁定时间。
- 所有金额通过四分之一凯利和统一风险限制。
- 不出现比分、半全场或 3 场以上串关。
- 每日、每月、单场和串关上限均有自动测试保护。
- 邮件图片包含当天方案、前一天结算、最近 30 天盈亏和历史累计摘要。
- 网站标题为“博弈预测看板”并保留完整盈亏历史。
- 所有历史方案、赔率、概率和结算均可追溯，重跑不会重复记账。
- 系统没有真实投注、支付或自动购买能力。
