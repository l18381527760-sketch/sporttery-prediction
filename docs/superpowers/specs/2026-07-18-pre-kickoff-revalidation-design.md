# 博弈预测看板：开赛前复核与临场模拟投注设计

## 1. 目标

将北京时间 14:00 的每日方案从“已经计入模拟账本的最终方案”改为“等待临场确认的初选方案”，并在每笔方案对应比赛开赛前完成两阶段复核：

1. 开赛前约 90 分钟进行内部价值筛查。
2. 开赛前约 30 分钟使用最新国内竞彩赔率进行最终确认。
3. 只有最终确认的方案才占用日/月预算并进入模拟盈亏账本。
4. 最终确认、撤销、赔率变化或金额变化通过额外邮件通知；同一方案最多发送一次临场通知。

本功能仍然只做模拟投注，不连接真实投注账户，不自动购买，不承诺盈利。

## 2. 已确认约束

- 北京时间 14:00 继续发送一封每日初选报告。
- 允许发送额外的临场方案变更邮件。
- 临场邮件按一次工作流运行合并，同一候选方案最多通知一次。
- 初选后不得增加新比赛、新玩法或替换候选项。
- 临场金额只能保持或降低，不能增加。
- 允许零投注日，也允许初选方案最终全部撤销。
- 最终赔率必须来自竞彩网；竞彩网临时不可用时，只允许使用完成比赛身份和玩法映射校验的中国足彩网数据。
- 专业博彩网站、交易市场和新闻数据只用于辅助分析，不得替换国内竞彩结算赔率。
- 无法可靠获取的阵容、伤停或天气信息必须显示为“未验证”，不得推测或伪造。
- 每日新增模拟投入不超过 500 元，每月不超过 5000 元；当月已实现亏损达到 5000 元后停止新增模拟投注。
- 现有历史账本、已锁方案和结算结果保持不变。
- 当前 value-v4 策略继续保持影子模式，不能因为加入临场复核而绕过激活审计。

## 3. 当前问题

### 3.1 监控快照不是实时赔率

当前 `odds-snapshot.yml` 每半小时运行一次，但 `capture_odds_snapshot.py` 默认读取当天不可变导入清单中的赔率。因此多个“监控快照”可能只是重复封装中午导入的相同数据，不能证明临场赔率发生或没有发生变化。

### 3.2 初选过早写入账本

当前决策工作流在约 13:30 生成整日方案、创建单一日期锁并写入模拟账本。晚间比赛尚未公布的阵容信息和后续赔率变化无法影响这笔模拟决策。

### 3.3 单一日期锁不适合错峰开赛

同一天比赛可能相隔数小时开赛。以一个时间戳锁定整日方案，会导致早场与晚场使用相同决策时点，无法证明每一笔方案都在其开赛前完成了临场确认。

### 3.4 自动化命令不一致

新版 `generate_betting_plan.py` 要求生成时提供带时区的 `--locked-at`，但 `daily-forecast.yml` 仍存在不带该参数的调用。下一次没有现有锁的基础预测可能失败。

### 3.5 邮件发送方需要保持唯一

Apps Script 已承担北京时间窗口、哈希校验和邮件去重职责。GitHub 的 `email-report.yml` 若仍有定时触发，会形成第二发送方，可能造成错误时间或重复邮件。

## 4. 方案比较

### 4.1 逐笔两阶段状态机（采用）

每个初选方案拥有稳定 `candidate_id`，分别记录 90 分钟筛查和 30 分钟最终确认，最终生成逐笔不可变执行凭证。

优点是适配错峰开赛、证据完整、不会重排整天预算，也能准确解释每笔撤销原因。缺点是需要新增状态文件、逐笔凭证和账本导入入口。

### 4.2 每次重建整日方案（不采用）

每次临场刷新都重新运行候选排序和资金分配。实现表面简单，但会增加新候选、替换比赛、移动其他比赛金额，并导致同一业务日出现多个互相矛盾的方案。

### 4.3 只做一次临场检查（不采用）

只在开赛前约 30 分钟检查。实现成本较低，但无法提前发现长期赔率恶化和数据问题，也缺少两个独立赛前观察点。

## 5. 总体架构

系统增加五个边界清晰的组件：

1. **初选方案生成器**：从不可变预测输入和 13:30 左右的最新国内赔率生成候选，不写正式模拟账本。
2. **实时国内赔率采集器**：直接访问当前竞彩网接口；必要时使用经过严格映射验证的中国足彩网数据，并写入追加式快照。
3. **临场复核状态机**：根据每个候选的最早开赛时间，执行 90 分钟筛查和 30 分钟最终确认。
4. **逐笔执行凭证与账本入口**：验证候选、实时快照、概率、金额和时间戳后，只追加最终确认的方案。
5. **报告与通知发布器**：构建网站、日报图片、临场更新状态，并由 Apps Script 发送去重邮件。

不可变的中午导入清单继续作为模型输入。实时赔率快照是独立的执行证据，不能覆盖中午清单，也不能反向修改预测概率。

## 6. 数据模型

### 6.1 初选方案

新增：

```text
output/provisional_betting_plan_YYYY-MM-DD.csv
output/provisional_shadow_plan_YYYY-MM-DD.csv
```

每行至少包含：

- `candidate_id`
- `report_date`
- `strategy_version`
- `match_id` 或规范化串关腿列表
- `team_a`、`team_b`、`kickoff_at`
- `market_type`、`market_line`、`selection`
- `initial_odds`、`initial_odds_source`、`initial_odds_captured_at_bjt`
- `raw_probability`、`calibrated_probability`、`conservative_probability`
- `minimum_acceptable_odds`
- `initial_expected_value`
- `initial_stake`
- `route`，取值为 `active` 或 `shadow`
- `candidate_payload_sha256`

`candidate_id` 只由业务日期、策略版本、规范化比赛/玩法身份和初选版本生成，不包含可变化的赔率或金额。初选文件通过规范化 CSV 字节计算 SHA-256，并写入状态文件。

### 6.2 实时赔率快照

新增目录：

```text
data/live_odds_snapshots/YYYY-MM-DD/
```

文件名格式：

```text
YYYYMMDDTHHMMSS+0800-<source>-<sha-prefix>.json
```

每个快照为 schema 1，至少保存：

- `report_date`
- `captured_at_bjt`
- `source`
- `fetch_mode: live`
- 原始响应摘要和采集代码提交号
- 比赛身份、销售状态、开赛时间和单关资格
- `had`、`hhad`、`ttg` 的当前赔率及来源记录标识
- 采集错误和身份映射诊断

快照采用排他创建，不允许覆盖已有路径。采集时必须重新访问数据源，禁止从不可变中午导入清单复制赔率。

### 6.3 临场状态

新增：

```text
output/revalidation_state_YYYY-MM-DD.json
```

状态文件为 schema 1，保存初选文件哈希、修订号、下次复核时间和每个候选的状态。候选状态只允许以下迁移：

```text
provisional -> screened -> confirmed
provisional -> cancelled
screened -> cancelled
```

终态 `confirmed` 和 `cancelled` 不可逆。任何重跑只能得到相同结果或验证现有终态，不能重新启用已撤销候选。

候选另有单调的账本副作用状态：`not_applicable`、`pending`、`ingested`。active 或 shadow 候选进入 `confirmed` 时先写为 `pending`；相应正式模拟账本或观察账本原子写入成功后才能变为 `ingested`。`cancelled` 使用 `not_applicable`。候选主状态不可回退，账本状态也只能从 `pending` 前进到 `ingested`。

### 6.4 复核检查点

每次检查写入独立凭证：

```text
output/revalidation_receipts/YYYY-MM-DD/<candidate_id>-t90.json
output/revalidation_receipts/YYYY-MM-DD/<candidate_id>-t30.json
```

凭证至少保存：

- schema 和候选身份
- 检查阶段、实际检查时间、距最早开赛分钟数
- 初选文件路径和哈希
- 实时赔率快照路径和哈希
- 当前赔率、最低可接受赔率和当前保守 EV
- 原始金额、重新计算金额和最终金额
- 国内赔率来源与原始记录标识
- 辅助证据可用性和已验证风险
- 决策 `pass`、`confirm` 或 `cancel`
- 机器可读原因代码和中文展示原因
- 规范化凭证 SHA-256

已有同名凭证只有在规范化内容完全相同时才允许重用，否则失败关闭。

### 6.5 临场更新状态与图片

新增公开索引、按业务日状态和不可变修订图片：

```text
web/revalidation-index.json
web/revalidation/YYYY-MM-DD/status.json
web/revalidation/YYYY-MM-DD/revision-<revision>-<change-digest-prefix>.png
```

`revalidation-index.json` 只列出最近两个业务日中仍有非终态候选或未发送更新的日期，并为每个日期保存状态 URL、修订号和状态文件 SHA-256。

按业务日 `status.json` 至少包含：

- `schema_version`
- `report_date`
- `revision`
- `changed_at_bjt`
- `change_digest`
- `changed_candidates`
- `next_revalidation_at_bjt`
- `all_candidates_terminal`
- `report_image_url`
- `report_image_sha256`
- `source_commit_sha`

`change_digest` 由本次新进入终态的候选规范化摘要计算。Apps Script 以 `report_date + change_digest` 进行临场邮件去重。

每次产生新的用户可见终态变化时创建一张新的修订图片；已有修订图片不可覆盖。这样即使北京时间午夜后新一天的 `web/daily-report.png` 已更新，前一业务日的临场邮件仍只能下载并校验与其状态绑定的旧业务日修订图片。

## 7. 时间与状态规则

### 7.1 基础预测与初选

- 12:15 后完成不可变数据导入、历史特征和概率预测。
- 13:30 后重新抓取一次实时国内赔率，并生成初选方案。
- 初选方案只允许包含距离开赛至少 60 分钟的比赛或串关。
- 距离开赛 60 至 105 分钟的候选，以初选实时快照同时生成 `t90` 凭证，并直接进入 `screened` 状态。
- 距离开赛超过 105 分钟的候选保持 `provisional`，等待独立 90 分钟筛查。
- 距离开赛不足 60 分钟、已开赛、开赛时间缺失或时区不明确的比赛不得进入初选。

### 7.2 90 分钟筛查

- 调度器在最早开赛前 105 分钟开始尝试，在距开赛超过 40 分钟时完成。
- 目标观察点是开赛前约 90 分钟；时间窗口容纳 GitHub 调度延迟。
- 必须有新的实时赔率快照，且采集时间晚于初选快照、早于比赛开赛。
- 通过后状态变为 `screened`，不写模拟账本，也不发送临场邮件。
- 赔率价值丧失、市场关闭、身份不一致、确认的重大风险或国内赔率不可验证时直接变为 `cancelled`。
- 未在窗口内完成筛查时使用原因 `t90_window_missed` 撤销，不允许在 30 分钟阶段补做。

### 7.3 30 分钟最终确认

- 调度器在距最早开赛 40 分钟时开始尝试，必须在距开赛大于 10 分钟时完成。
- 目标观察点是开赛前约 30 分钟。
- 只有 `screened` 候选可以确认。
- 必须使用晚于 90 分钟凭证的新实时赔率快照。
- 符合全部条件时状态变为 `confirmed`，生成最终执行凭证并尝试写入模拟账本。
- 任何条件不满足时状态变为 `cancelled`。
- 距开赛不超过 10 分钟仍未完成时使用原因 `t30_window_missed` 撤销。

### 7.4 串关

- 串关以最早开赛的一腿确定两个复核窗口。
- 90 分钟和 30 分钟快照都必须同时包含全部腿，且全部比赛尚未开赛。
- 每一腿分别通过市场开放、身份、赔率来源、最低赔率和正价值检查。
- 任一腿失败即撤销整个串关，不替换腿，也不降级为单关。

### 7.5 跨午夜业务日

复核器不能只处理北京时间当天。每次运行扫描尚有非终态候选的最近两个竞彩业务日，并依据候选内的绝对 `kickoff_at` 决定是否到期。

Apps Script 的每日初选仍按当前北京时间业务日发送；临场更新以 `report_date + change_digest` 标识，因此前一业务日的欧洲深夜比赛在午夜后仍可完成复核和发送一次更新。

## 8. 价值与金额复核

临场复核固定使用初选时保存的 `conservative_probability`，不因观察到后续赔率而重新训练或选择概率，避免后见偏差。

对当前国内十进制赔率 `o` 和保守概率 `p`，其中候选生成阶段已验证 `0 < p < 1`：

```text
current_ev = p * o - 1
minimum_acceptable_odds = (1 + configured_minimum_ev) / p
```

候选必须同时满足：

- 当前赔率有效且大于 1。
- 当前赔率不低于初选保存的 `minimum_acceptable_odds`。
- 当前 EV 达到该玩法配置的最低 EV。
- 国内来源、比赛身份、玩法、让球/总进球参数和单关资格保持一致。
- 市场仍在销售且比赛尚未开赛。

最终金额重新使用当前赔率计算保守四分之一凯利金额，然后应用：

```text
final_stake = floor_to_2_yuan(min(initial_stake, recalculated_stake, remaining_caps))
```

最终金额不能超过初选金额。金额低于 2 元时撤销。账本的日/月/单场/串关上限按最终确认顺序原子检查；额度不足时降低金额，降低后不足 2 元则撤销。

## 9. 辅助证据

现有专业市场热度和赔率变化数据可以参与风险说明，但不能成为国内赔率凭证。

阵容、伤停和天气采用三态：

- `verified_clear`：指定可信来源已检查且没有触发配置风险。
- `verified_risk`：指定可信来源明确记录重大缺阵、比赛条件异常或其他配置风险，撤销候选。
- `unavailable`：无法可靠验证，报告显示“未验证”；不推测具体球员或影响。

本阶段不以不稳定网页文本推断阵容，不因缺少阵容数据自动声称“阵容正常”。赔率、比赛身份和销售状态仍是最终确认的强制数据。

## 10. 账本与历史兼容

### 10.1 新账本证据版本

`betting_ledger.py` 增加逐笔复核导入路径。新确认行至少增加：

- `evidence_type: revalidation_receipt`
- `candidate_id`
- `candidate_payload_sha256`
- `t90_receipt_path`、`t90_receipt_sha256`
- `t30_receipt_path`、`t30_receipt_sha256`
- `live_odds_snapshot_path`、`live_odds_snapshot_sha256`
- `final_confirmed_at_bjt`

导入时重新验证两个凭证、快照、候选身份、最终赔率、金额、开赛时间和预算。任何哈希、身份或时间不一致都拒绝整笔导入。

同一次运行有多笔候选进入最终窗口时，按最早开赛时间、初选排名、`candidate_id` 排序后依次检查预算，保证重跑和并发恢复得到相同结果。

### 10.2 路由规则

- `route=active` 且最终确认的候选进入正式模拟盈亏账本。
- `route=shadow` 的候选只进入观察账本，不占用正式模拟预算。
- value-v4 在激活审计通过前仍保持 `shadow`。
- 现有日期锁和旧账本行继续由原验证路径处理，不迁移、不重算。

上述路由还受临场复核自身的 `mode` 控制：`mode=shadow` 时，所有确认结果只写入独立演练账本，不改变正式模拟盈亏或观察账本；`mode=active` 时才应用 active/ shadow 路由规则。临场复核模式与 value-v4 激活模式是两个独立开关，任何一个都不能隐式改变另一个。

### 10.3 幂等性

稳定 `bet_id` 由业务日期、`candidate_id` 和最终凭证哈希生成。重复工作流只能验证和重用同一行，不能重复投入或重复结算。

工作流先发布终态和不可变凭证，再导入账本，最后把 `ledger_status` 从 `pending` 更新为 `ingested`。若账本导入失败，重跑必须扫描所有 `confirmed + pending` 候选并重试相同凭证；在 `ingested` 之前不得发布包含该确认项的用户可见 `change_digest` 或临场邮件。

## 11. 自动化工作流

### 11.1 `daily-forecast.yml`

职责缩小为：

- 导入或验证当天不可变清单。
- 构建历史特征与预测。
- 生成基础网站数据，但不创建正式方案锁、不写模拟账本。
- 修复所有生成命令与当前 CLI 参数的契约不一致。

### 11.2 `draw-alert-refresh.yml`

作为 13:30 初选阶段：

- 重新抓取实时国内赔率。
- 生成初选 active/shadow 方案和初始复核状态。
- 生成平局预警与辅助市场证据。
- 构建标记为“初选”的网站和日报图片。
- 发布可供 14:00 邮件使用的报告状态。

该阶段不创建旧式整日方案锁，也不调用正式模拟账本导入。

### 11.3 `pre-kickoff-revalidation.yml`

新增工作流，支持 `workflow_dispatch` 并保留每 10 分钟 cron 作为后备。每次运行：

1. 扫描最近两个业务日的非终态候选。
2. 若没有到期候选，成功退出且不写文件。
3. 对到期候选重新抓取实时国内赔率。
4. 原子执行 90 分钟或 30 分钟状态迁移。
5. 为最终确认项写逐笔凭证并导入相应账本。
6. 重建网站、业务日临场状态和不可变修订图片，再原子更新 `revalidation-index.json`。
7. 只在文件实际变化时提交 `data output web`。

所有写仓库的工作流继续共用 `sporttery-repository` 并发组。

### 11.4 `odds-snapshot.yml`

保留每 30 分钟监控用途，但改为真正访问实时数据源。它不得读取中午清单中的赔率来冒充新快照，也不得修改候选状态或账本。

### 11.5 `noon-settlement.yml`

只结算已经最终确认并进入账本的行。初选、`screened` 和 `cancelled` 候选不产生投入、返还或盈亏。

### 11.6 `email-report.yml`

移除定时触发，只保留受控手工诊断入口，或明确禁止生产发送。Apps Script 继续是唯一生产邮件发送方。

## 12. Apps Script 调度与邮件

### 12.1 初选邮件

现有 14:00 至 18:00 初选报告发送窗口和图片哈希校验继续有效。初选邮件必须明确显示：

- “初选方案，尚未计入模拟盈亏”。
- 每笔最早开赛时间和下一复核时间。
- 最终方案可能确认、降额或撤销。

初选发送成功后记录独立状态 `LAST_INITIAL_SENT_DATE` 和图片哈希。升级时只把格式有效且日期为当天的旧 `LAST_SENT_DATE` 视为兼容初选发送状态，不复制旧图片哈希到新的临场去重状态。

### 12.2 临场工作流调度

Apps Script 每 10 分钟读取 `revalidation-index.json`，校验索引中的状态文件哈希，再读取最多两个业务日的状态。当任一状态的 `next_revalidation_at_bjt` 到期且存在非终态候选时，触发 `pre-kickoff-revalidation.yml`。调度使用独立冷却键，不能阻塞下一业务日基础预测。

GitHub cron 是后备路径；Apps Script dispatch 是主要可靠路径。两条路径依靠候选状态机和不可变凭证保持幂等。

### 12.3 临场邮件

当 `report_date + change_digest` 尚未发送、对应业务日的初选邮件已经成功发送、状态指定的不可变修订图片哈希校验通过时，发送一封临场更新邮件。邮件按一次工作流运行合并所有新终态候选，列出：

- 比赛和玩法。
- 最终确认或撤销。
- 初选赔率与临场赔率。
- 初选金额与最终金额。
- 当前 EV。
- 主要原因。

同一 `candidate_id` 只能进入一次临场邮件。状态从 `screened` 到 `confirmed/cancelled` 才对用户可见，90 分钟内部通过不会单独发信。即使最终赔率和金额恰好未变，`screened -> confirmed` 仍属于一次最终状态确认，并进入该业务日下一封合并临场邮件。

临场邮件允许在 18:00 后和午夜后发送，直到该候选进入终态；它不受初选报告 18:00 失败通知截止限制。若当天初选邮件从未成功发送，则不发送孤立的临场更新邮件，只更新网站和状态。

Apps Script 保存有界的最近 30 个业务日已发送 `change_digest`，避免 Script Properties 无限增长。

## 13. 网站和图片

网站与日报图片从同一结构化状态构建。每笔候选显示以下状态之一：

- 初选待复核
- 90 分钟筛查通过
- 临场确认
- 临场降额
- 已撤销

正式模拟盈亏表只统计“临场确认”且已经成功进入账本的 active 路由行。初选总额单独显示为“暂定金额”，不能与已投入混合。

取消原因使用短中文展示，完整机器原因保留在凭证和网站详情中。

## 14. 失败关闭规则

以下情况必须撤销受影响候选，不得使用旧赔率继续：

- 实时国内赔率抓取失败或响应结构无效。
- 比赛身份、开赛时间、玩法或让球参数无法对应。
- 市场停止销售或单关资格失效。
- 当前赔率低于最低可接受赔率。
- 当前 EV 低于配置门槛。
- 90 分钟或 30 分钟窗口错过。
- 初选、快照、凭证或状态文件哈希不一致。
- 候选比赛已经开赛。
- 最终预算检查失败且无法降至合法金额。

单个候选失败不应阻止其他身份独立的候选完成复核，但任何共享串关腿失败会撤销整个串关。

若状态文件损坏或出现不允许的状态回退，工作流整体失败且不写账本、不发送临场邮件。

## 15. 测试策略

所有行为变更采用测试驱动开发。测试分为：

### 15.1 实时采集

- 证明实时模式实际调用数据源，不读取中午赔率作为当前赔率。
- 竞彩网成功、严格中国足彩网回退、身份映射失败和市场关闭。
- 已开赛比赛过滤、时区验证、快照排他写入和冲突拒绝。

### 15.2 状态机

- 合法状态迁移和所有非法回退。
- 60 分钟初选下限、90/30 分钟窗口、错过窗口撤销。
- 重跑幂等、跨午夜扫描和串关最早开赛规则。
- 初选后不得新增候选或恢复已撤销候选。

### 15.3 价值与预算

- 最低可接受赔率、当前 EV 和四分之一凯利重算。
- 金额只能保持或降低，2 元取整，低于 2 元撤销。
- 日/月/单场/串关限制与月度止损。
- active 和 shadow 路由隔离。

### 15.4 凭证与账本

- 候选、两次快照和两次凭证哈希验证。
- 仅 confirmed active 行进入正式模拟账本。
- 重复导入不重复投入。
- 任何赛后快照、时间倒序或篡改都失败。
- 旧日期锁和历史账本继续通过原有测试。

### 15.5 工作流与邮件

- 12:15 命令契约正确。
- 初选阶段不写正式账本。
- 复核工作流只在到期时采集和提交。
- GitHub 定时邮件被禁用。
- Apps Script 初选、临场 dispatch、跨午夜索引、按业务日图片哈希校验、去重和失败重试。
- 同一候选最多一次临场通知，一次运行内变化合并。

### 15.6 全量回归

- Python 全量测试。
- Apps Script Node 测试。
- 工作流静态契约测试。
- Python 编译检查。
- `git diff --check`。

## 16. 上线与回滚

### 16.1 影子上线

第一阶段只生成初选、复核状态、逐笔凭证和 shadow 观察结果，不改变正式模拟账本。至少完成一次跨错峰开赛的完整业务日并人工核对：

- 实时快照确实来自不同采集时点。
- 两个窗口和开赛时间顺序正确。
- 撤销原因准确。
- 邮件没有重复。
- 午夜后仍能处理前一业务日。

### 16.2 正式模拟切换

影子演练通过后，以明确切换日期启用逐笔 active 账本导入。切换只影响此后新候选，不迁移历史。

### 16.3 回滚

回滚时停止 `pre-kickoff-revalidation.yml` 和 Apps Script 对应 dispatch，保留所有已生成凭证和已确认账本行。不得恢复把初选直接当成最终投入的旧行为；若临场系统不可用，正确降级是当天零新增模拟投注，并发送数据未就绪说明。

## 17. 验收标准

功能完成必须同时满足：

1. 14:00 报告明确为初选，且初选金额不进入模拟盈亏。
2. 每笔最终确认都有两个有序赛前检查点，或由 60 至 105 分钟初选快照合法承担第一检查点。
3. 最终赔率来自新的实时国内赔率快照，而不是中午清单副本。
4. 最终金额不超过初选金额，并通过全部预算限制。
5. 错过窗口或数据不可验证时自动撤销。
6. 同一候选最多一封临场邮件，重复运行不重复发送。
7. 午夜后仍可复核前一业务日尚未开赛的候选。
8. value-v4 保持影子模式，激活门槛不被绕过。
9. 旧账本、旧日期锁和历史盈亏不被重写。
10. 全量自动测试、编译检查和工作流契约测试全部通过。
